# Registro de Decisões — NÓS

> Cada decisão importante entra aqui com o porquê. Antes de re-discutir algo, leia a entrada. Formato: D-nº · data · decisão · motivo.

- **D-01 · 2026-07-14 · Nome: NÓS.** Contrário de "Jogador Nº 1"; trocadilho triplo (nós = coletivo, nós = nós do grafo git, nós = rede). Decidido com o Tiago.
- **D-02 · 2026-07-14 · Repo público dedicado (`brigsd/nos`).** Actions ilimitado grátis, Pages grátis, contribuição comunitária. Decidido com o Tiago.
- **D-03 · 2026-07-14 · 100% GitHub.** Repo = banco, Actions = servidor, Pages = cliente, issues = comandos. É a tese do projeto; sem servidores externos.
- **D-04 · 2026-07-14 · Tick em lote, 1 commit por tick.** Escala com o tempo, não com jogadores; evita corrida de commits e histórico inchado.
- **D-05 · 2026-07-14 · Combate por turnos autoritativo (v2).** Resolvido pela engine no tick com seed determinística; cliente exibe replay. À prova de trapaça por arquitetura. PvP assíncrono.
- **D-06 · 2026-07-14 · TypeScript em engine e cliente.** Tipos compartilhados eliminam divergência entre servidor e apresentação.
- **D-07 · 2026-07-14 · Pages via artifact deploy.** Site publicado não entra no histórico git; repositório permanece leve indefinidamente.
- **D-08 · 2026-07-14 · Arte: pixel art 16×16, paleta Resurrect 64.** Estilo alcançável por sprites-como-código + packs CC0 (Kenney etc.); o art-reviewer valida visualmente.
- **D-09 · 2026-07-14 · NPCs v1/v2 sem LLM em runtime.** Árvores de comportamento + falas roteirizadas: determinístico, grátis e sem depender de API key. LLM é extensão opcional futura.
- **D-10 · 2026-07-14 · Estrutura de agentes + docs de continuidade.** Sugestão do Tiago: qualidade garantida por revisores especializados e progresso garantido por documentação de estado, não por memória de conversa.
- **D-11 · 2026-07-14 · Tick híbrido: relógio + evento.** Batida do mundo por cron (1h) E processamento de comandos disparado por evento de issue (~30–60s de latência percebida). `concurrency` do Actions aglutina comandos simultâneos num lote — 1 commit por execução segue valendo.
- **D-12 · 2026-07-14 · UI otimista no cliente.** Ação aparece na hora como "pendente" (fantasma) e reconcilia quando o tick confirma; movimento de terceiros interpolado entre batidas. A latência real fica invisível.
- **D-13 · 2026-07-14 · Login GitHub no site via OAuth device flow (v2).** Site estático autentica sem servidor e cria comandos via API por baixo dos panos — jogar sem sair do jogo. Fallback permanente: issue manual.
- **D-14 · 2026-07-14 · Notificações via resposta de issue.** O tick responde o comando na issue do jogador; o app oficial do GitHub entrega como push no celular. Notificação nativa, custo zero.
- **D-15 · 2026-07-14 · NPCs com IA via GitHub Models (v2.5).** Atualiza o D-09: o GitHub Models oferece inferência gratuita dentro do Actions com o token nativo — Nativos podem decidir/conversar com IA sem serviço externo e sem API key. Comportamento-base continua procedural (fallback se o rate limit apertar).
- **D-16 · 2026-07-14 · A Crônica auto-escrita.** Cada batida registra eventos; um job periódico os transforma em capítulos legíveis publicados no site. O mundo acumula história auditável (git) sem escritor humano.
- **D-17 · 2026-07-14 · Federação de mundos (v3).** Cada repositório de criador = um planeta (tick e Pages próprios); portais são um protocolo (registro de endereços + passaporte de avatar). O NÓS é a rede e O Coração, o mundo-origem.
- **D-18 · 2026-07-14 · Quests que rendem achievements reais (v3).** Ações de jogo geram PRs reais mesclados automaticamente (validados por CI) — jogador ganha Pull Shark/Pair Extraordinaire jogando. Gancho de divulgação único.
- **D-19 · 2026-07-14 · Tick auto-corretivo + snapshots.** O cron do GitHub atrasa às vezes: o tick calcula batidas perdidas e compensa. Snapshots periódicos do mundo como releases (backup e arqueologia).
- **D-20 · 2026-07-14 · Sem dinheiro. Ponto.** Decisão do Tiago: nenhuma forma de monetização, nunca — sem anúncios, sem patrocínio, sem recompensas de valor real, sem conversão do Pulso. Elimina por construção os riscos de fraude/bots por lucro, o jurídico de promoções e o atrito com o uso aceitável do GitHub. O Pulso tem valor social e de jogo, apenas.
- **D-22 · 2026-07-15 · "O Registro": o eu oficial é sólido, o eu local é fantasma.** Decisão do Tiago, nascida de um acidente feliz: o cliente renderiza dois avatares do jogador — o oficial (estado do mundo) e o local (movimento otimista, D-12). Em vez de esconder um, assumimos: o avatar do mundo (o *Registro*) é desenhado sólido; o avatar local (a *intenção*) é translúcido (alpha 0.4). Só o que o Pulso gravou é real; você é intenção até a batida te escrever na Crônica. É o D-12 virando poesia visível e o tema do NÓS (o nó entre quem você é e quem o mundo registrou) virando mecânica.
- **D-21 · 2026-07-14 · Economia federada: ledger central + leitura mútua (v3).** Saldo de ₱ vive só no Coração (`ledger/`); mundos federados registram eventos num `outbox.json` público que o tick do hub lê a cada batida (federação *pull-based* via raw URLs — sem tokens nem webhooks entre repos). Avatar presente em 1 mundo por vez (check-in/check-out sequencial no hub ⇒ gasto duplo impossível). Emissão de ₱ por mundo limitada por cota conforme nível de confiança no `federation.json`; mundos com engine modificada têm economia local livre, mas não emitem ₱ central.
- **D-23 · 2026-07-15 · O Coração é A Fábrica — mundo pacífico, sem combate.** Decisão do Tiago. O Coração vira o hub industrial do metaverso: jogadores trazem materiais/recursos padronizados dos outros mundos e FABRICAM aqui, via receitas e requisitos, tudo o que os outros mundos usam (veículos, ferramentas, equipamentos, tecnologia). Ofícios com minijogos próprios (metalurgia, culinária, química, engenharia, marcenaria, medicina, robótica…), introduzidos em fases. Combate NÃO existe n'O Coração — pode existir em outros mundos (v3), atualizando o D-05 (que passa a valer para mundos de combate, não para o hub). Por quê: (a) identidade — nenhum MMO tem um hub-fábrica como mundo central, e "o Coração bombeia recursos para o corpo" torna a metáfora literal; (b) passa no filtro anti-genérico ("por que isso só funciona aqui?": porque o hub federado com ledger central, D-21, é único da nossa arquitetura); (c) escopo — foco total no ecossistema de fabricação em vez de dividir esforço com combate. Regra de corte adotada junto: toda mecânica nova precisa responder "por que isso só funciona no NÓS?" — vetados de nascença XP/níveis como progressão central, grind de repetição e poder individual sem teto.
- **D-24 · 2026-07-15 · Autonomia total do coder + política de agentes.** Decisão do Tiago: o coder desenvolve sem depender do ideador — decide, implementa, revisa e mescla os trilhos técnicos (portais, fluidez P2P/polling, janela WebGL, motor de fabricação, login GitHub). Agentes: **codadores em Sonnet 5, revisores em Opus 4.8**; despacho a cargo do coder (proteger a janela de contexto). Documentação de continuidade também a cargo do coder. Com o ideador ficam apenas: design dos minijogos de ofício e divulgação/lançamento. Decisões de produto novas continuam sendo levadas a ele.

Transparência obrigatória: P2P opt-in com aviso de exposição de IP. (d) **Auth = GitHub OAuth device flow** (D-13 confirmado; Google descartado — identidade do jogador é o login GitHub). - **D-26 · 2026-07-15 · Janela WebGL (PixiJS) como opt-in; Canvas2D padrão e fallback permanente.** Decisão do coder com evidência (PR #41, `docs/R3_COMPARATIVO_RENDER.md` na branch `claude/r3-webgl-comparativo`). Motivo: o ganho é VISUAL (água por shader, bloom/CRT de custo fixo — o caminho pro alvo Eastward/Octopath do D-25e), não performance (sandbox sem GPU tornou os FPS não-conclusivos; validar em GPU real durante a migração). Custo ~76 kB gzip via import() dinâmico, pago só por quem liga. Migração incremental: interface Renderer → renderer-webgl.ts lazy → flag localStorage (padrão Canvas) → validação em dispositivo real. O contrato de mundo permanece intocado — janelas são descartáveis por arquitetura.
- **D-27 · 2026-07-15 · Portais fase 1: travessia sem sair do site, protocolo de registro, visitar é só-leitura (R6, atualiza o D-17).** Decisão do coder (trilho autônomo, D-24), documentando fechamentos concretos do design em aberto do D-17 ("registro de endereços + passaporte de avatar"). (a) **Registro é um arquivo (`worlds/registry.json`), não um serviço**: array de `{id, name, worldUrl?, clientHint?, status, descriptionPtBR}` — `worldUrl` aceita tanto uma URL raw absoluta (mundo federado de verdade, outro repositório) quanto um caminho relativo (mundo local, ex. `world/heart.json`/`worlds/atrio.json`); um repositório federado entra via PR adicionando sua própria entrada, sem tocar engine/. (b) **`worlds/` (plural) ≠ `world/` (singular)**: o primeiro é o hall do protocolo + mundos estáticos de teste; o segundo continua sendo só o estado oficial e tick-gerenciado d'O Coração — nunca colidem. (c) **"Visitar" nesta fase é deliberadamente só-leitura e sem estado no servidor**: fetch + valida (reaproveitando `isPlausibleWorld` de `live.ts`) + troca o mundo em tela; nenhum check-in/check-out, nenhum "onde meu avatar está" gravado em lugar nenhum — check-in/check-out sequencial (D-21) e agir em mundo remoto ficam para quando a federação de verdade chegar (`docs/PORTALS_PROTOCOL.md`, seção própria). Os links/botões de ação de Comércio/Nativos/Oficinas somem durante a visita (sempre mirariam O Coração, então mostrá-los sobre outro mundo seria enganoso, não só incompleto). (d) **O pulso ao vivo d'O Coração pausa durante a visita** (`liveHandle.stop()`) e um refetch fresco + novo polling substituem ao voltar — nunca faz sentido gastar ciclos de API por um mundo que não está na tela. (e) **O marco do portal no mapa é 100% client-side, nunca `world.machines`/`world.natives`** — nasce do registro, não do motor; achado de QA descartou a borda oeste (a coluna de painéis do HUD cobre a esquerda da tela inteira no zoom padrão, e a Câmera fica travada/centralizada nesse zoom — não dá pra arrastar até revelar o tile) em favor da borda leste, sempre visível e clicável na carga inicial. (f) **O Átrio** (`worlds/atrio.json`) é o primeiro mundo de teste: 32×32, gerado deterministicamente (mesmo `Rng` do motor, seed própria), sem Núcleo, proporcionalmente mais água (um lago autocontido, não um rio de borda a borda — assinatura visual distinta) e mais ruínas (um salão retangular deliberado, não blobs orgânicos) que O Coração, para a travessia ser inconfundível à primeira vista.
- **D-28 · 2026-07-15 · Protocolo de sinalização P2P sobre GitHub (R7 "Fluidez A", fecha o D-25c).** Decisão do coder (trilho autônomo, D-24), fechando o design de como dois navegadores se encontram usando só uma issue do GitHub como quadro de avisos. (a) **A issue "P2P: sala d'O Coração" é achada-ou-criada uma vez por sessão** via search API (título exato; em empate por corrida, o número mais baixo vence — convergência determinística sem lock) e nunca carrega o label `comando` nem o prefixo `Comando:` no título — verificado linha a linha contra `.github/workflows/tick.yml`: o job só dispara em `issues: [opened]` (nunca `issue_comment`, então comentários de sinalização são 100% inertes) e seu próprio `if:`/filtro `jq` exigem um dos dois marcadores, que esta issue nunca tem. (b) **Convites (oferta/resposta SDP) são comentários com um bloco ```json cercado**, `{v, kind, from, to, sdp}`; `from` é sempre cruzado contra o autor REAL do comentário (nunca confiado do corpo sozinho — bloqueia falsificação de identidade). ICE é "vanilla" (não-trickle): cada lado espera terminar de coletar candidatos (ou um teto de segurança) antes de mandar UM SDP completo, porque o polling a cada 3–5s tornaria trickle-ICE por comentário impraticavelmente lento. (c) **Presença nasce da própria oferta**: cada par, uma vez, publica uma oferta broadcast (`to:'*'`); quem vê uma oferta alheia fresca só responde se `meuLogin > oferta.from` (comparação de string) — critério determinístico sem coordenação que garante no máximo uma conexão por par (nunca duas correndo em paralelo). Limite documentado: com 3+ jogadores simultâneos a malha pode ficar incompleta (a oferta de cada um só fecha com quem a responder primeiro) — aceitável para o caso principal do produto (dois jogadores), republicar oferta após ser reivindicada fica para depois. (d) **Canal de dados não-confiável de propósito** (`ordered:false, maxRetransmits:0`) — posição obsoleta não vale a pena retransmitir; o heartbeat periódico dobra como reenvio de posição (não só um ping vazio), rede de segurança barata contra o próprio canal não-confiável perder o pacote inicial. (e) **Encerramento**: ao desligar, os próprios comentários viram "(encerrado)" (edição best-effort, nunca bloqueia o teardown); o polling GitHub para de vez ao conectar (`stop when connected`, retomado só se o par cair).
