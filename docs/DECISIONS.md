- **D-27 · 2026-07-15 · Portais fase 1: travessia sem sair do site, protocolo de registro, visitar é só-leitura (R6, atualiza o D-17).** Decisão do coder (trilho autônomo, D-24), documentando fechamentos concretos do design em aberto do D-17 ("registro de endereços + passaporte de avatar"). (a) **Registro é um arquivo (`worlds/registry.json`), não um serviço**: array de `{id, name, worldUrl?, clientHint?, status, descriptionPtBR}` — `worldUrl` aceita tanto uma URL raw absoluta (mundo federado de verdade, outro repositório) quanto um caminho relativo (mundo local, ex. `world/heart.json`/`worlds/atrio.json`); um repositório federado entra via PR adicionando sua própria entrada, sem tocar engine/. (b) **`worlds/` (plural) ≠ `world/` (singular)**: o primeiro é o hall do protocolo + mundos estáticos de teste; o segundo continua sendo só o estado oficial e tick-gerenciado d'O Coração — nunca colidem. (c) **"Visitar" nesta fase é deliberadamente só-leitura e sem estado no servidor**: fetch + valida (reaproveitando `isPlausibleWorld` de `live.ts`) + troca o mundo em tela; nenhum check-in/check-out, nenhum "onde meu avatar está" gravado em lugar nenhum — check-in/check-out sequencial (D-21) e agir em mundo remoto ficam para quando a federação de verdade chegar (`docs/PORTALS_PROTOCOL.md`, seção própria). (d) **O pulso ao vivo d'O Coração pausa durante a visita** (`liveHandle.stop()`) e um refetch fresco + novo polling substituem ao voltar — nunca faz sentido gastar ciclos de API por um mundo que não está na tela. (e) **O marco do portal no mapa é 100% client-side, nunca `world.machines`/`world.natives`** — nasce do registro, não do motor; achado de QA descartou a borda oeste (a coluna de painéis do HUD cobre a esquerda da tela inteira no zoom padrão, e a Câmera fica travada/centralizada nesse zoom — não dá pra arrastar até revelar o tile) em favor da borda leste, sempre visível e clicável na carga inicial. - **D-28 · 2026-07-15 · Protocolo de sinalização P2P sobre GitHub (R7 "Fluidez A", fecha o D-25c).** Decisão do coder (trilho autônomo, D-24), fechando o design de como dois navegadores se encontram usando só uma issue do GitHub como quadro de avisos. (a) **A issue "P2P: sala d'O Coração" é achada-ou-criada uma vez por sessão** via search API (título exato; em empate por corrida, o número mais baixo vence — convergência determinística sem lock) e nunca carrega o label `comando` nem o prefixo `Comando:` no título — verificado linha a linha contra `.github/workflows/tick.yml`: o job só dispara em `issues: [opened]` (nunca `issue_comment`, então comentários de sinalização são 100% inertes) e seu próprio `if:`/filtro `jq` exigem um dos dois marcadores, que esta issue nunca tem. (b) **Convites (oferta/resposta SDP) são comentários com um bloco ```json cercado**, `{v, kind, from, to, sdp}`; `from` é sempre cruzado contra o autor REAL do comentário (nunca confiado do corpo sozinho — bloqueia falsificação de identidade). ICE é "vanilla" (não-trickle): cada lado espera terminar de coletar candidatos (ou um teto de segurança) antes de mandar UM SDP completo, porque o polling a cada 3–5s tornaria trickle-ICE por comentário impraticavelmente lento. (c) **Presença nasce da própria oferta**: cada par, uma vez, publica uma oferta broadcast (`to:'*'`); quem vê uma oferta alheia fresca só responde se `meuLogin > oferta.from` (comparação de string) — critério determinístico sem coordenação que garante no máximo uma conexão por par (nunca duas correndo em paralelo). Limite documentado: com 3+ jogadores simultâneos a malha pode ficar incompleta (a oferta de cada um só fecha com quem a responder primeiro) — aceitável para o caso principal do produto (dois jogadores), republicar oferta após ser reivindicada fica para depois. (d) **Canal de dados não-confiável de propósito** (`ordered:false, maxRetransmits:0`) — posição obsoleta não vale a pena retransmitir; o heartbeat periódico dobra como reenvio de posição (não só um ping vazio), rede de segurança barata contra o próprio canal não-confiável perder o pacote inicial. (e) **Encerramento**: ao desligar, os próprios comentários são **apagados** (DELETE best-effort, nunca bloqueia o teardown) — editar para "(encerrado)", como na primeira versão, deixaria o SDP com o IP do jogador legível para sempre no histórico público de edições do GitHub (achado de review); o polling GitHub para de vez ao conectar (`stop when connected`, retomado só se o par cair). (f) **Anti-spam de sinalização** (achado de review): falha de conexão (canal nunca abriu — ex. NAT simétrico dos dois lados, onde só-STUN nunca conecta e TURN é proibido pelo D-25c) entra em backoff exponencial por login (30s→8min), desistência após 4 falhas seguidas e teto duro de 20 ofertas por sessão — sem isso, o par re-ofertava/re-respondia para sempre na issue pública, comendo a cota de escrita que os comandos reais do jogador usam. Saída limpa (canal chegou a abrir) re-anuncia na hora, sem backoff.
